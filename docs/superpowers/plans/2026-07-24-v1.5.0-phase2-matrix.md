# v1.5.0 Phase 2 вЂ” KeyГ—Context Matrix Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Reconcile every keybinding's actual behavior with the help bar, `:help`, and README across all interaction contexts, fixing confirmed divergences вЂ” the v1.5.0 phase-2 release-gate line.

**Architecture:** Discovery is done token-consciously per the execution spec (`docs/superpowers/specs/2026-07-24-v1.5.0-phase2-matrix-execution.md`): enumerate both axes inline to build a fixed grid, fan out to five self-contained parallel `Agent` calls (surface-partitioned) plus one completeness-critic, synthesize into a matrix ledger, verify divergences inline, triage with the owner in batches, then land fixes repro-first one commit each. **No `Workflow`** is used вЂ” direct `Agent` calls only, so there is no resumable state to cache-miss.

**Tech Stack:** Go 1.x, `rivo/tview` on `gdamore/tcell`; standard `testing` package; `gofmt`/`go vet`/`staticcheck` gate.

## Global Constraints

- Full gate every commit: `go test ./...`, `go vet ./...`, `staticcheck ./...`, `go build ./...`; `gofmt` clean.
- All work on `ai-workspace`; never commit/merge to `main`.
- Every change appends a dated `## YYYY-MM-DD вЂ” Title` entry to `log.md`, newest at top, one heading per entry.
- Any user-visible behavior change updates `README.md` and `:help` (`internal/ui/help.go`) in the same increment; any design change updates `main.md` in the same increment.
- Every code fix is repro-first: a failing test demonstrating the bug, then the fix, then green вЂ” one commit per fix.
- Discovery uses direct parallel `Agent` calls, never a `Workflow`. Every agent prompt is fully self-contained (file targets + doc text embedded); no agent consumes another agent's output.
- Only `internal/ui` imports tview/tcell. New selectable widgets carry `selectionStyle` (see CLAUDE.md guardrail).
- Deliverable ledger lives at `docs/audit/specdiff/MATRIX.md`, mirroring phase 1's `CLAIMS.md`.

```go
func TestModalMotionKeyTranslatesLetters(t *testing.T) {
	cases := []struct {
		in   rune
		want tcell.Key
	}{
		{'j', tcell.KeyDown},
		{'k', tcell.KeyUp},
		{'h', tcell.KeyLeft},
		{'l', tcell.KeyRight},
	}
	for _, c := range cases {
		got := modalMotionKey(tcell.NewEventKey(tcell.KeyRune, c.in, tcell.ModNone))
		if got.Key() != c.want {
			t.Errorf("modalMotionKey(%q) = %v, want %v", c.in, got.Key(), c.want)
		}
	}
}

func TestModalMotionKeyPassesThroughNonMotion(t *testing.T) {
	// A real arrow and a non-motion rune must pass through unchanged.
	arrow := tcell.NewEventKey(tcell.KeyDown, 0, tcell.ModNone)
	if got := modalMotionKey(arrow); got != arrow {
		t.Errorf("arrow key should pass through unchanged")
	}
	q := tcell.NewEventKey(tcell.KeyRune, 'q', tcell.ModNone)
	if got := modalMotionKey(q); got != q {
		t.Errorf("'q' should pass through unchanged")
	}
}
```

- [ ] **Step 3: Add the helper in `keys.go` (after `motionArrow`, ~line 164)**

```go
// modalMotionKey lets a modal list share the app-wide h/j/k/l motion idiom.
// A modal short-circuits globalKeys (see app.go), so the letter motions never
// reach motionArrow there вЂ” translate them locally via the same table. Real
// arrow keys already work and any non-motion event passes through untouched.
func modalMotionKey(ev *tcell.EventKey) *tcell.EventKey {
	if arrow, isLetter, ok := motionArrow(ev); ok && isLetter {
		return tcell.NewEventKey(arrow, 0, tcell.ModNone)
	}
	return ev
}
```
