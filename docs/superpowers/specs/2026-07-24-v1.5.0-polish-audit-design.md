# v1.5.0 — Polishing & Auditing: Design

> Owner-approved design for LazyPlanner v1.5.0 (settled 2026-07-24). v1.5.0 is a consolidation
> release — the last planned release for the foreseeable future — with no new features beyond two
> approved gap-closers. On approval of this spec, the scope is written into `main.md` as the
> `### v1.5.0` Build Plan subsection before implementation begins (the standing planning rule).

---

## Goals

1. **Spec conformance, exhaustively verified**: every behavioral claim in the docs is checked
   against the program (or explicitly dispositioned by the owner), in both directions — broken
   promises *and* undocumented behavior.
2. **UI/keymap self-consistency**: keys, modes, hints, and docs agree everywhere; behavior that
   would confuse a user is reconciled.
3. **Deep-audit coverage of the post-pass-18 feature surface**: at least one hardening pass over
   the never-audited v1.2.0–v1.4.0 code, then best-effort passes toward convergence.
4. **Docs current** at release: main.md, README, `:help`, and the audit ledgers all truthful.

## Non-goals

- **No new features** beyond the two gap-closers below. Configurable keybindings, persistent
  trash, conflict keep-both-as-separate-items, and full `sync-collection` delta sync stay on the
  deferred list.
- **No convergence gate on the release**: the two-consecutive-no-HIGH audit streak is pursued
  best-effort; hardening continues post-release as v1.5.x patches regardless.
- **Pi hardware smoke test** stays an owner-side, optional release-checklist item (unauditable
  headlessly).

## Owner-settled decisions (2026-07-24)

| Question | Decision |
|---|---|
| Scope seed | Systemic sweeps, priority-ordered: spec↔program gaps → UI/keymap inconsistencies → deep audit if time allows |
| Spec-gap depth | Exhaustive, claim-by-claim, **both directions** (broken promises + doc drift) |
| UX gap-closers | **Both** in scope: agenda-board click-to-select and Detail-pane accordion |
| Audit commitment | **Minimum one pass**, then best-effort toward the no-HIGH streak; cadence continues post-release |
| Spec vs code disagreements | **Triage each with the owner**: findings batched with a recommended resolution (fix code vs fix doc); owner signs off before fixes land |
| Execution style | Hybrid: durable claim inventory + parallel agent fan-out for verification; **speed and maximum parallelism encouraged** |

## Step 0 — Fix the known `moveSubtreeOps` data-loss gap

Before any sweeping: `moveSubtreeOps`'s source-side rewrite (`internal/ui/yankpaste.go`) commits
via a bare `store.Put` on an existing resource instead of `store.PutIfUnchanged` — the
version-check-invariant gap flagged in `docs/audit/COVERAGE.md` (predates v1.4.0; now also reached
by the multi-root `pasteMultiRoot` path).

- Repro-first: a failing test demonstrating a concurrent sync pull clobbered during a cross-list
  move, then route the write through `PutIfUnchanged`, then the test goes green and stays as the
  regression guard.
- Update the `COVERAGE.md` flag in the same increment. One commit, full gate.

## Phase 1 — Exhaustive spec↔program diff (both directions)

### Deliverable: the claim inventory

`docs/audit/specdiff/CLAIMS.md` — a living ledger (the `COVERAGE.md` pattern) holding **every
behavioral claim** extracted from:

- `main.md`'s design sections (UI Design, Settled Decisions, and the behavioral content of the
  Build Plan records),
- `README.md` (usage, keybindings table, syncing, configuration),
- `:help` (`internal/ui/help.go`).

Each claim carries: an id, the source doc + line, the claim text, a status
(`unverified → holds / code-diverges / doc-stale`), and evidence (`file:line` or a repro). The
inventory makes "exhaustive" inspectable: an unverified claim is visible, never silently sampled
away.

### Forward verification (spec → program)

Parallel agent fan-out — claims are independent, so they verify concurrently (one workflow per doc
region). Every reported divergence is **adversarially re-verified** with a runnable repro or exact
code-path evidence before it reaches triage (the audit protocol's evidence-over-verdict rule; a
workflow's own summary is unverified until checked).

### Reverse direction (program → spec)

Parallel agents sweep the code for user-visible behavior the docs don't describe: key handlers and
chords, `:` commands, flash/status messages, config options, status-bar states, mouse behavior.
Output: doc-gap findings ("the program does X; no doc says so").

### Triage and fixes

- Findings go to the owner **in batches**, each with a recommended resolution (fix code vs fix
  doc) — the owner is the design authority; shipped behavior sometimes beats what was written.
- After sign-off: code fixes land **repro-first, one commit each, full gate**; doc fixes may batch
  into grouped commits. The inventory's statuses update as findings resolve.

### Seed finding (found during design)

main.md's Pane-sizing section says the `+`/`-` accordion collapses "the side panels **and
Detail**", while its own Future-versions list says the accordion "only collapses the overview
column today". The spec contradicts itself; the code's actual behavior decides which line is
`doc-stale` (and gap-closer B makes the richer claim true regardless).

## Phase 2 — UI/keymap consistency sweep + two gap-closers

### Deliverable: the key×context matrix

Every key/chord crossed with every interaction context — NORMAL/DRILL/GRAB/SELECT/RESIZE × task
tree / month grid / week-day grid / agenda / the three overview panels / forms (NORMAL + DRILL) /
modals — recording what the key **actually does** there, and what the **help bar**, **`:help`**,
and **README** each say. Built by agent fan-out reading `keys.go`, `selection.go`, `grab*.go`,
`caretForm`, and the render/hints paths.

A finding is any cell where:

- the same key means something different across contexts *unintentionally* (deliberate context
  sensitivity like `+`/`-` zoom-vs-accordion is fine — but must be documented),
- a modal layer's swallow contract leaks (the SELECT bare-`0` class),
- the mode badge, help bar, or hint line misstates what movement keys act on,
- counts (`3j`) behave inconsistently across surfaces that accept motion,
- `Esc` unwinds a different number of layers than the mode-nesting design promises,
- the three doc surfaces disagree with the behavior or each other.

Fixes follow the same triage → repro-first → one-commit-each pipeline as phase 1.

### Gap-closer A — agenda-board click-to-select

The center agenda board is the one surface where the mouse does nothing; worse, a double-click
there edits whatever was *already selected* (the pass-16 accepted gap, now closed):

- Position→item hit-testing on the board; **single click selects** the item under the cursor,
  **double-click edits** it — matching every other view's mouse contract.
- Follows the `treeNodeAtY` precedent: public tview APIs only, re-target before acting.
- Tests: headless hit-testing tables covering **both sides of every boundary** (first/last row,
  one-past-last, empty board, scrolled board — the canary guardrail), plus a double-click
  edits-the-row-under-cursor regression mirroring `TestTreeNodeAtY*`.

### Gap-closer B — Detail-pane accordion

`+`/`-` (outside the week/day hour-zoom context) collapses/restores the **Detail pane** together
with the overview column, making main.md's existing accordion wording true:

- Reuses the existing `Flex.ResizeItem` machinery and the accordion's current toggle state; the
  restored Detail width keeps honoring the remembered state-file width (never the config file).
- Agenda mode already hides Detail; the accordion must not fight that (collapse state and
  mode-driven hiding stay independent).
- Tests: accordion toggle round-trip including Detail, week/day zoom precedence unchanged,
  display-stress states extended for the new geometry (the standing rule for new draw behavior).

## Phase 3 — Deep audit (minimum one pass, then best-effort)

Run the `/audit` hardening workflow at least once, targeted at the least-audited surfaces:

1. **SELECT mode + bulk ops + bulk grab** (`internal/ui`) — never audited; built 2026-07-23.
2. **v1.3.0 recurrence-rewrite primitives + Repeat/Custom UI** — the write-side additions landed
   after the last recurrence audit (pass 14).
3. **v1.2.0 quick-add grammar extensions** (ranges, recurrence, relative dates, location,
   warnings) — landed after the last parser audit (pass 14).
4. **Sync-core reconcile-vs-concurrent-pull matrix** beyond the `CommitPush` window
   (tombstone/keep-both races) — re-warmed by pass 18, not cleared.

Standard rules of engagement apply (`docs/audit/PROTOCOL.md`): coverage-first targeting,
adversarial verification with repros, mutation canaries, bounded residual-risk reporting, ledger
updated afterwards. Further passes toward the two-consecutive-no-HIGH streak run only as time
allows.

---

## Sequencing (maximum parallelism)

1. **Step 0** lands first — it touches code the sweeps will read.
2. **Phase 1 and phase 2 discovery run concurrently**: claim extraction and matrix building read
   different doc surfaces and write no code. The two **gap-closers** are independent of both
   sweeps and can be built while verification agents run.
3. **Fixes land serially per triage batch** (parallel fixes on one branch collide); batches from
   the two phases interleave as they're signed off.
4. **Phase 3 starts after phases 1–2's code fixes settle**, so the audit sees final code.

## Cross-cutting rules (standing practice, restated)

- Full gate (`go test`, `go vet`, `staticcheck`, `go build`) every commit; `log.md` entry per
  change; spec/README updated in the same increment as any behavior change.
- Every code fix repro-first with a regression test; `COVERAGE.md` and the claim inventory kept
  current.
- All work on `ai-workspace`; merge/tag/release is the owner's alone.

## Release criteria

- Step 0 done; claim inventory fully dispositioned (no `unverified` rows); key×context matrix
  reconciled; both gap-closers shipped with tests; ≥1 audit pass completed with its findings
  fixed or owner-accepted; all docs current.
- Owner reviews, merges to `main`, and tags `v1.5.0`.
